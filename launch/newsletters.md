# Newsletters

Short, format-matched pitches for OSS/AI newsletters. These curate heavily — lead with the
one-sentence "why it's interesting," not a feature list. **Confirm each outlet's current
submission method before sending** (forms and addresses change); links below are starting
points. Keep the honest caveats in — these editors notice overclaiming.

Reusable one-liner:

> A developer had Anthropic's frontier model (Fable 5) write behavioral-discipline skills
> for the smaller Opus 4.8, then test-drove them on the target model with a TDD-for-docs loop
> — pressure scenarios run on real subagents, failures captured verbatim, fixes verified.
> Transcripts included; the write-up is candid that only scenario-level flips are proven, not
> daily gains. MIT. github.com/DizzyMii/fable-skills

---

### 1. TLDR AI
Tip via their "submit" channel / reply-to on the daily issue. Format: one punchy sentence +
link, written in their clipped style.
> **Frontier model writes discipline skills for the smaller one, tested on the target.** A dev
> distilled Fable 5's behavioral contract into six Claude Code skills for Opus 4.8, then TDD'd
> them on real Opus subagents — verbatim failures, verified flips, transcripts in the repo. [link]
→ https://tldr.tech/ai (look for the "submit a tip" / sponsor-vs-editorial distinction)

### 2. Console.dev (dev tools / OSS)
Console reviews developer tools and OSS. Use their tool-submission form. They favor a clear
"what it is / who it's for / why it's notable."
> **What:** Six Claude Code skills hardening Opus 4.8's behavior (claims, scope, reporting).
> **Why notable:** authored by the frontier model, then pressure-tested on the target model
> with full transcripts — including the finding that a skill's *example* out-teaches its rules.
> **License:** MIT. [link]
→ https://console.dev/submit/

### 3. Changelog (News / podcast)
Submit via Changelog News submission. Angle: the method is the story (TDD-for-docs; example
beats rules), not the product.
→ https://changelog.com/news (submit a story)

### 4. Ben's Bites / AI newsletters with a "tools" section
One sentence + link, emphasis on the novel build method and the honesty of the test record.
> A dev had Anthropic's frontier model write the discipline rules for its smaller sibling,
> then ran TDD-for-docs on the target model — and published every failing transcript. [link]

### 5. Pointer.io / TLDR-style eng digests
Best fit is the write-up (`launch/article.md`), not the repo alone — these link essays. Pitch
the article URL once it's published, with the example-beats-rules lesson as the hook.

---

**General rules:** one pitch per outlet, no follow-up spam, disclose you're the author, and
match the outlet's voice. Newsletters reward a genuinely novel angle — here it's "the frontier
model wrote it and I tested it on the target," plus "I published the failures." Don't pitch a
productivity claim the repo doesn't make.
